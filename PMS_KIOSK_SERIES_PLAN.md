# PMS x Kiosk 系列發文企劃

更新日期：2026-06-22

## 系列定位

`PMS x Kiosk：打造更順的旅客入住體驗` 是 `旅館生活科技` 系列底下的一條實用支線，聚焦中小型旅宿如何把 PMS、Kiosk、自助發卡機、門禁設備、AI agent 與邊緣運算接成一條完整的旅客入住流程。

這個系列不是單純介紹硬體，也不是把 PMS 寫成後台系統說明書。核心問題是：旅客從訂房完成到真正進房，中間有哪些資料、權限與現場任務需要被整理好？Kiosk 如何從一台螢幕，變成現場任務的執行點？

## 讀者承諾

讀者看完這個系列，應該能理解：

- PMS 在智慧入住裡扮演什麼角色。
- Kiosk / iPad 自助櫃檯不只是前台畫面，而是旅客任務入口。
- 自助發卡機、手機憑證、QR Code 與門禁設備如何和 PMS 資料同步。
- 為什麼邊緣運算與斷網備援是旅宿現場的關鍵。
- 中小旅宿如何用漸進式方式導入，而不是一次重做整套系統。

## 系列語氣

- 實用、清楚、偏經營者視角，但保留 Cellbedell Blog 的生活科技語氣。
- 每篇都從旅客或櫃台的一個場景開始，不從技術名詞開始。
- 技術要翻成「現場會發生什麼事」。
- 不把 AI、Kiosk 或 PMS 神化；每篇都要說清楚限制、例外與真人介入點。

## 建議發文節奏

建議一週 3 篇，避免連續每日硬發導致文章太像產品說明。可採週一、三、五主文，週二或週四補短文/社群摘要。

- 週一：主架構文章，建立系列脈絡。
- 週三：單一流程拆解，讓讀者理解導入細節。
- 週五：檢查清單或案例型文章，方便保存與分享。

### 1. 2026-06-23

**主標題**：PMS 如何接上自助 Kiosk？從訂單資料到現場報到

**SEO 角度**：PMS kiosk integration, hotel self check-in kiosk, 旅館自助入住系統

**核心問題**：PMS 是資料核心，Kiosk 是旅客入口；兩者中間要怎麼把訂單、房號、入住狀態與權限接起來？

**文章重點**：

- 旅客看到的是自助報到畫面，背後其實是 PMS 資料流。
- PMS 提供訂單、房型、房號、入住日期、付款與房務狀態。
- Kiosk 負責資料確認、旅客互動、任務觸發與現場引導。
- Cellbedell 的定位：Kiosk / iPad / 發卡機 / 門禁設備透過 API 與邊緣運算協同作動。

**圖片需求**：

- PMS 到 Kiosk 的資料流概念圖。
- 自助櫃檯或發卡機場景圖。

**內部連結**：

- 連到 `旅館自助入住一定要買大型 Kiosk 嗎？`

### 2. 2026-06-25

**主標題**：旅客抵達前，哪些資料應該先被整理好？

**SEO 角度**：pre check-in hotel, hotel guest data, smart check-in workflow

**核心問題**：自助入住順不順，往往取決於旅客抵達前資料是否已整理好。

**文章重點**：

- 訂單資料、身份資料、抵達時間、付款或押金。
- 特殊需求、房型偏好、同行旅客與語言偏好。
- 哪些資料可以預先收集，哪些需要現場確認。
- 資料整理不好，Kiosk 只會把櫃台問題搬到螢幕上。
- Cellbedell 的 pre check-in 流程：旅客訂房後先在手機或電腦完成 check-in、付款與金鑰取得；抵達現場後由邊緣設備完成驗證，不需要即時仰賴雲端連線；Kiosk 確認後發放對應房卡。

**圖片需求**：

- 抵達前資料 checklist 圖。
- Pre check-in -> 付款與金鑰 -> 現場邊緣驗證 -> Kiosk 發卡流程圖。

**內部連結**：

- 連到 `智慧入住的下一步`

### 5. 2026-07-02

**主標題**：斷網時怎麼辦？智慧入住不能只靠雲端

**SEO 角度**：offline hotel access control, edge computing hotel check-in

**核心問題**：智慧入住最怕關鍵時刻無法作動，斷網備援是旅宿現場必須被設計進去的能力。

**文章重點**：

- 哪些任務可以離線執行：已授權發卡、進門判斷、基本紀錄。
- 哪些任務需要恢復連線後同步：紀錄、報表、異常通知。
- 邊緣運算是自主化現場流程，不是只做展示用。

- 需要真人介入的場景：改房、退款、特殊需求、外國旅客、家庭入住。
- 好的自助入住應該讓服務更有溫度，而不是更冷。
